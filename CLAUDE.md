# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm run setup        # First-time init: install deps, generate Prisma client, run migrations
npm run dev          # Start dev server (Next.js + Turbopack) at http://localhost:3000
npm run build        # Production build
npm run lint         # ESLint
npm test             # Run all Vitest tests
npm run db:reset     # Reset SQLite database (destructive)
```

Run a single test file: `npx vitest run src/lib/__tests__/file-system.test.ts`

The dev server requires `NODE_OPTIONS='--require ./node-compat.cjs'` (already wired into `package.json` scripts). Do not remove this flag.

## Environment

`.env` accepts an optional `ANTHROPIC_API_KEY`. Without it, the app uses a mock LLM (`MockLanguageModel` in `src/lib/provider.ts`) that returns static example components instead of calling Claude.

## Architecture

UIGen is a Next.js 15 App Router application. The core loop is: user describes a component in chat â†’ streaming API call to Claude â†’ Claude uses tool calls to write/edit files in an in-memory virtual file system â†’ preview iframe renders the result in real time.

### Virtual File System (`src/lib/file-system.ts`)

All generated code lives in a `VirtualFileSystem` (in-memory `Map<string, FileNode>`). Nothing is written to disk. The VFS is serialized to JSON and stored as a string column in SQLite (`Project.data`) when the user saves. It is deserialized back on load.

Every project requires `/App.jsx` as the entry point. Local imports use the `@/` alias which is resolved at preview time via an import map.

### AI Tool System (`src/lib/tools/`)

Claude is given two tools:
- `str_replace_editor` â€” create and edit files (`view`, `create`, `str_replace`, `insert`, `undo_edit` commands)
- `file_manager` â€” rename and delete files

Tool execution is handled client-side in `FileSystemContext` (`src/lib/contexts/file-system-context.tsx`). When the streaming response delivers tool calls, `handleToolCall` updates the VFS and triggers a preview refresh.

### Chat + File System State (`src/lib/contexts/`)

`ChatContext` wraps Vercel AI SDK's `useChat` and holds message history. `FileSystemContext` holds the live VFS state. Both are provided at the root in `src/app/main-content.tsx`. They communicate through `onToolCall` callbacks â€” the chat stream fires tool calls, and the file system context executes them.

### Preview Rendering (`src/lib/transform/jsx-transformer.ts`)

The `PreviewFrame` component (`src/components/preview/PreviewFrame.tsx`) renders an `<iframe>` whose HTML is generated by `jsx-transformer.ts`:
1. Babel standalone transforms JSX/TSX â†’ JS at runtime (in-browser, no build step)
2. An ES module import map resolves `react`, `react-dom`, and `@/` paths
3. Local modules are served as `blob:` URLs injected into the import map
4. The generated HTML document is set as `srcdoc` on the iframe

### API Route (`src/app/api/chat/route.ts`)

`POST /api/chat` accepts `{ messages, files, projectId }`. It rebuilds the VFS from `files`, calls Claude (or mock) with the system prompt and tools, streams the response back as a `DataStreamResponse`, and on completion persists the updated project to Prisma if the user is authenticated.

### Authentication (`src/lib/auth.ts`, `src/actions/`)

JWT-based auth stored in an httpOnly cookie (`session`). Server actions in `src/actions/` handle sign-up, sign-in, sign-out, and project CRUD. Anonymous users can create projects; projects are linked to a user on sign-in.

### Database (`prisma/schema.prisma`)

SQLite via Prisma. Refer to `prisma/schema.prisma` for the authoritative data model. Two models: `User` and `Project`. `Project.messages` and `Project.data` are JSON-serialized strings holding the full chat history and VFS state respectively.

## Key Conventions

- TypeScript path alias `@/*` maps to `./src/*` (configured in `tsconfig.json`)
- UI primitives come from Radix UI, styled with Tailwind v4; shared components live in `src/components/ui/`
- Tests use Vitest + React Testing Library + JSDOM; test files sit in `__tests__/` subdirectories next to the code they test
